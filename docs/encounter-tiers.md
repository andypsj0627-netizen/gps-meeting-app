# 조우 등급제 + 시공간 겹침 설계 (2026-07-14 확정)

두 기능(실시간 조우, 시공간 겹침)을 하나의 시스템으로 설계한다.
겹침/조우가 이벤트를 **생산**하고, 등급제가 그 누적을 **의미(해금 수준)**로 바꾼다.

```
이벤트 생산                    누적 원장                의미 변환
실시간 조우(구현됨)   ─┐
시간차 겹침(신규)     ─┼→  encounters/{쌍ID}  →  등급 파생  →  해금 수준·피드 카드
패턴 겹침(신규)       ─┘   (횟수·날짜·장소)
```

## 핵심 설계 결정

### 1. 원장은 하나, 이벤트 소스는 둘
실시간 조우와 시공간 겹침을 별개 시스템으로 나누지 않는다. 나누면 "어제 실시간으로
만난 사람이 겹침 피드에서는 잠김"인 모순이 생긴다. 사용자에게는 "이 사람과 나의
인연 히스토리" 하나로 보인다.

### 2. 등급(sticky)과 근접 상태(휘발)의 2축 분리
7/13의 "멀어지면 재잠금" 결정과 등급제의 sticky함은 역할 분리로 해소한다:
- **등급** — 누적 이력에서 파생, 내려가지 않음(감쇠 제외) → 프로필 해금 수준 담당
- **근접 상태** — 현재 60m/100m 히스테리시스(기존 `activeUserIds`) → "지금 근처"
  표시와 운명 연출만 담당

기존 재잠금 로직은 폐기가 아니라 역할이 좁아진다.

### 3. 대화는 항상 상호 호감 게이트 뒤에
등급/조우로 열리는 것은 **프로필 깊이와 호감 표시 버튼까지**다. 채팅은 양쪽 다
호감을 눌렀을 때만 열린다. 등급이 최고여도 예외 없음 — 데이팅 앱의 괴롭힘 벡터를
차단하는 안전 게이트이며, 두 기능 공통의 최종 관문이다.

## 등급 사다리

| 등급 | 조건 | 해금되는 것 |
|------|------|------------|
| 잠김 | — | 회색 마커, 탭 무반응 |
| 스침 | 겹침·조우 1회 | 흐린 프로필: 이니셜, 나이대("20대"), 거리 밴드 |
| 인연 | **서로 다른 날** 2~3회 | 이름·나이·한줄소개 + 맥락 카드("이번 주 3번째 스침") |
| 운명 | 실시간 조우 발생 | 사진 + "지금 근처" 펄스 연출 |
| 연결 | 상호 호감(쌍방 좋아요) | 채팅 |

- 등급 상승은 **서로 다른 날 기준** — 하루 종일 따라다니며 등급을 올리는
  스토킹형 게이밍 차단. 같은 시간 버킷 내 중복은 1회로 카운트
- **감쇠**: 마지막 조우 후 30일 경과 시 피드에서 자연 소멸 (위치 이력 TTL 30일과 정합)
- 실시간 조우는 유일한 해금 수단에서 → 희소한 "운명" 트리거로 재포지셔닝

## 시공간 겹침의 정의

**같은 geohash 셀(정밀도 7, ~150m) + 같은 시간 버킷(1시간)에 둘 다 기록이 있으면 겹침 1건.**

- "같은 순간"일 필요 없음 — "어제 19시대, 같은 카페 골목"이면 성립
- 장소만 겹침(언젠가 같은 카페)은 **아님** — 시간 조건이 없으면 매칭이 무의미하게
  많아지고 프라이버시 부담이 커진다
- 반복 패턴("매주 화요일 같은 헬스장")은 이 정의 위에서 요일·시간대 집계로 잡는다

## 데이터 모델 (위치 데이터 모델 문서와 정합 — README 참조)

```
locationHistory/{uid}/points/{id}   ← 내 이력: cell(geohash7), bucket(1h),
                                       dayOfWeek/hourOfDay(패턴용), TTL 30일
cellVisits/{cell_bucket}/{uid}      ← 역색인: "이 셀·이 시간대에 누가 있었나"
encounters/{uid1_uid2}              ← 쌍 원장: count, distinctDays, firstAt/lastAt,
                                       realtimeCount, 상위 셀 집계, tier(파생 캐시)
likes/{uid}/{targetUid}             ← 호감. 쌍방 존재 시 "연결"
```

**겹침 판정은 역색인 방식**: 위치 기록 시 `cellVisits`에도 쓰고, 그 문서의 기존
방문자들과 겹침 이벤트를 생성한다. 이력 전체 조인이 아니라 셀 단위 만남 지점에서
즉시 발견되므로 쿼리가 싸다. 이상적으로는 Cloud Function이 처리(정밀 좌표 비노출)
하지만 Functions는 Blaze 요금제가 필요 — 초기엔 클라이언트가 자기 방문 시점에
역색인을 읽고 기록하는 방식으로 시작 가능.

## 프라이버시 가드레일 (기능의 성패를 가름)

- 저장·노출 모두 **150m 셀 + 1시간 버킷 해상도** — 상대에게 보이는 건
  "어제 저녁, ○○동 부근" 수준. 정확한 시각·좌표는 어디에도 노출 안 됨
- **쌍방 노출 원칙**: 내가 상대 피드에 보이면 상대도 내 피드에 보임 (일방 관찰 불가)
- 옵트아웃 가능, 단 끄면 본인도 겹침 피드 사용 불가 (상호주의)
- 차단 시 쌍 원장·피드에서 즉시 제거
- 위치 이력 저장이므로 **위치정보법상 위치기반서비스사업 신고 대상 확정** (Phase 4)

## 노출 UX

지도와 별개의 **"스침 피드" 탭**. 카드 = 등급별 해금된 프로필 + 맥락 한 줄
("이번 주 3번째 · 매번 화요일 저녁"). 맥락 문장이 그대로 아이스브레이커가 된다.
장소명 표기는 v1에서 "○○동 부근" 수준(역지오코딩).

## 구현 경로

- **Phase A (지금 착수 가능, Firestore 불필요)**: 등급제를 시뮬레이션으로 개발·검증.
  `EncounterDetector` 이벤트를 로컬 원장(순수 Dart)에 누적 → 등급 파생 → 마커
  색/바텀시트 해금 수준 연동. 가상 사용자 10명으로 크롬에서 바로 확인 가능
- **Phase B (Phase 3와 함께)**: 위치 이력 + 역색인 + 스침 피드 — 확정된 위치 데이터
  모델(geohash+시간 버킷, TTL 30일)이 그대로 기반
- **Phase C**: 패턴 겹침(요일·시간대 반복), 감쇠, 운명 연출 강화

## 열린 결정

- 셀 150m / 버킷 1시간은 출발값 — 도심 밀도 실측 후 튜닝
- "인연" 승급 기준: 서로 다른 날 2회 vs 3회
- 감쇠 방식: 피드 소멸만 vs 등급 자체 하락
