## 모델 역할 분담: Advisor / Worker

메인 세션은 Advisor다. 판단에 집중하고, 구현 노동은 Worker에게 위임한다.

**Advisor(메인 세션)가 직접 하는 일**
- 요구사항 분석, 작업 분해, 설계 결정
- Worker에게 줄 작업 브리프 작성
- 결과 검증: diff 직접 확인, 테스트 직접 실행
- 최종 커밋 승인, 사용자 보고

**Worker(Opus 서브에이전트)에게 위임하는 일**
- 코드 작성과 수정, 테스트 작성 등 구현 작업 전부
- Agent 도구로 위임하고 model은 `opus`를 지정한다
- 서로 독립적인 작업은 병렬로 위임한다

**브리프 기준**
- Advisor가 이미 파악한 컨텍스트를 담아 Worker가 재탐색하지 않게 한다
- 파일 경로, 프로젝트 컨벤션, 알려진 함정, 완료 기준(통과해야 할 테스트)을 포함한다

**경계**
- Worker의 완료 보고를 그대로 믿지 않는다. diff와 테스트로 직접 확인한 뒤 승인한다
- 검증 실패는 수정 브리프로 재위임한다. 직접 수정은 사소한 마무리에만 허용된다
- 한두 줄 수정처럼 위임 오버헤드가 더 큰 작업은 Advisor가 직접 처리해도 된다

## 피처 개발 워크플로우

새 기능을 추가할 때 아래 순서를 따른다. 각 단계의 주체(Advisor/Worker)는 위 역할 분담 섹션을 따른다.

### 1단계 — 탐색 (필요 시)
기존 코드와 연관된 작업이라면 Advisor가 먼저 맥락을 파악한다.
- 목적: 관련 파일, 패턴, 의존성 파악

### 2단계 — 설계
Advisor가 구현 전 청사진을 먼저 만든다. 파일 목록, 데이터 흐름, 컴포넌트 구조 포함.
- 산출물: 구현할 파일 목록 + 각 역할 + 빌드 순서
- **사용자 승인 후 3단계로 진행**

### 3단계 — 구현
설계 청사진을 바탕으로 Worker에게 구현을 위임한다.
- dart analyze 훅이 자동으로 각 파일 저장 시 실행됨

### 4단계 — 테스트 작성
구현한 로직에 대한 단위/위젯 테스트를 Worker에게 위임해 작성시킨다.
- 위치: `test/` 디렉토리, 파일명은 `*_test.dart`
- 테스트 없이 다음 단계로 넘어가지 않는다

### 5단계 — flutter test (하드 게이트)
테스트를 실제로 실행해서 pass 여부를 기계가 판단한다.
- 커맨드: `flutter test`
- **실패 시 커밋 불가** — 세션 종료 시 Stop 훅이 자동으로 강제함
- AI 리뷰보다 이 단계가 더 신뢰도 높음

### 6단계 — 코드 리뷰 (보조)
AI가 놓칠 수 있는 설계 관점의 문제를 점검한다.
- 커맨드: `/code-review`
- 체크 항목: 아키텍처 일관성, 보안, 가독성
- flutter test를 통과했어도 이 단계는 생략하지 않는다

### 7단계 — 커밋
모든 게이트 통과 후 커밋한다.
- 커맨드: `/commit`
- 커밋 메시지는 `feat:` / `fix:` / `refactor:` 접두어 사용

### 검증 계층 (신뢰도 순)
| 단계 | 도구 | 판단자 | 거짓말 가능성 |
|------|------|--------|--------------|
| 정적 분석 | dart analyze | 기계 | 없음 |
| 테스트 실행 | flutter test | 기계 | 없음 |
| 코드 리뷰 | /code-review | AI | 있음 (보조로만 사용) |

### 워크플로우 요약
```
탐색(선택) → 설계 → [승인] → 구현 → 테스트 작성 → flutter test → 코드 리뷰 → 커밋
```
